# Phase 14B Investigation — `/api/chat` 500 / Timeout Between Assistify and Ollama

**Mode:** Investigation only. No models changed. No switch to Qwen 3B. Evidence gathered before any fix.
**Runtime model (confirmed):** `qwen2.5:7b` — `config.py:178` default and the model actually resident in VRAM (`/api/ps`).
**GPU:** single 8 GB device (shared by embedder, reranker, Whisper, XTTS, and the 7B LLM).
**Evidence artifacts:** `scripts/phase14b_ollama_timing.py`, `logs/phase14b_ollama_timing_evidence.log`.

---

## Executive summary

The 500 / timeout is **not** generated by Ollama failing to produce output, and it is **not** caused by aiohttp or WebSocket transport. Once the model is resident, Ollama generates reliably at **~53–72 tok/s** and finishes in **< 17 s** for every profile tested.

The timeouts are produced by **Assistify forcing Ollama to reload the 7B model**, because Assistify sends a **different `num_ctx` per query type** while the warmup pinned the model at `num_ctx=3072`. Ollama reloads the model whenever `num_ctx` changes. On the 8 GB GPU each reload costs **5–15 s in isolation, and longer under VRAM contention**. That reload happens *before the first token*, so it races directly against Assistify's first-token / pipeline timeouts. When the reload (plus a long `num_predict=900` decode) outlasts those client-side caps, Assistify abandons the socket, runs a non-stream fallback, and/or retries — producing the **~45 s** terminations and the **~92 s** (two stacked attempts) responses.

**Origin of the timeout:** Assistify's client-side timeout policy in `streaming_service.py` / `rag_service.py` / `routing.py`, triggered by Assistify's own inconsistent `num_ctx`. Ollama, aiohttp, and the WebSocket layer are where the symptom surfaces, not the root cause.

## 3. Root cause: `num_ctx` mismatch → model reload

`num_ctx` is a **load-time** parameter in Ollama. Changing it evicts and reloads the model. Assistify uses different values per query type:

| Query type | `num_ctx` | `num_predict` | Source |
|------------|-----------|---------------|--------|
| Warmup preload | **3072** | — | `assistify_rag_server.py:2755` |
| default / fact / simple / list | 3072 | 96–180 | `streaming_service.py:4204`, `rag_service.py:1925` |
| `extreme_summary` | **4096** | 400 | `streaming_service.py:4194` |
| `multi_source_synthesis` | **6144** | 520 | `streaming_service.py:4199` |
| `executive_memo` | **6144** | 900 | `streaming_service.py:4184` |
| `quiz_generation` | **6144** | 850 | `streaming_service.py:4189` |
| `call_llm_with_context` | 3072 | 220 | `routing.py:1994` |

The code even warns about this:

```4222:4223:backend/services/streaming_service.py
        "options": {
            "num_ctx": _llm_num_ctx,     # IMPORTANT: Must match warmup exactly to avoid VRAM reloading
```

```2752:2756:backend/assistify_rag_server.py
        # IMPORTANT: num_ctx must match chat requests exactly, otherwise Ollama
        # reloads the model on every chat call (causing 8-second cold-start delays).
        "options": {
            "num_ctx": 3072,
```

Because warmup pins `3072` but memo/quiz/summary/synthesis send `4096`/`6144`, **any of those queries reloads the model**, and the **next** default query reloads it back. Alternating query types reloads on every turn.

### Measured proof (`logs/phase14b_ollama_timing_evidence.log`)

| Run | Profile (num_ctx) | `load_duration` | first-token (wall) | decode | total |
|-----|-------------------|-----------------|--------------------|--------|-------|
| Cold start | general (3072) | **15.00 s** | 15.91 s | 54.7 tok/s | 16.59 s |
| Warm, same ctx | general (3072) | 0.13 s | **0.17 s** | 72.4 tok/s | 0.87 s |
| Warm, same ctx | executive_memo (6144) ×2 | 0.12 s | **0.16 s** | ~67 tok/s | 6.9–11.3 s |
| **ctx switch 6144→3072** | general (3072) | **9.81 s** | **10.54 s** | 66.8 tok/s | 11.21 s |
| **Alternating 6144/3072/6144** | — | **5.29 / 10.12 / 5.44 s** | 6.2 / 11.0 / 6.3 s | ~53 tok/s | <17 s |

Same `num_ctx` twice ⇒ `load_duration ≈ 0.1 s`. Any `num_ctx` change ⇒ reload of **5–15 s** before the first token. This is deterministic and reproducible.

## 4. Why requests terminate at ~45 s, and why retries produce ~92 s

The first-token delay (model reload) collides with the client-side caps. Several independent paths converge on 45 s:

- **WebSocket stream path:** the `asyncio.gather(producer, consumer)` wrapper times out at `STREAM_TOTAL_TIMEOUT_S = 30 s` (`streaming_service.py:4907-4914`); it then runs the non-stream fallback at `LLM_FALLBACK_TOTAL_TIMEOUT_S = 15 s` (`:4920`). **30 + 15 = 45 s.**
- **memo/quiz/extreme stream path:** first-token timeout is literally **45 s** (`:4477`), raising `TIMEOUT_FIRST_TOKEN` → non-stream fallback.
- **Grounded generation path:** `call_llm_with_context` aborts on `sock_read=45` / `total=60` (`routing.py:2012`) and returns the "no match" refusal.
- **HTTP `/query` sync fallback:** `requests.post(timeout=45)` (`rag_service.py:2049`).

**~92 s = two stacked 45 s attempts.** The clearest match is the HTTP sync fallback loop:

```2057:2076:backend/services/rag_service.py
        for attempt in range(max_retries):
            status, text_body = await asyncio.get_event_loop().run_in_executor(None, _sync_post, LLM_URL, payload, 45)
            ...
            await asyncio.sleep(0.5 * (attempt + 1))
```

attempt 1 (45 s) + `sleep 0.5` + attempt 2 (45 s) ≈ **90.5 s ≈ 92 s**. The async path retries similarly (`rag_service.py:2014-2029`: on `asyncio.TimeoutError` it closes the socket, sleeps `1·(attempt+1)` s, rebuilds the session, retries), so a 45 s-class failure followed by a second attempt lands in the same ~92 s band.

**Retries make it worse, not better:** when aiohttp/`asyncio.wait_for` times out client-side, the POST is abandoned but **Ollama is not told to stop** (and `keep_alive=-1`). Ollama keeps loading/generating in the background. The retry then often arrives with the *same or different* `num_ctx`, so it either waits on the still-busy model or triggers **another reload**. So the retries are both a *consequence* of the first-token delay and an *amplifier* of total latency.

## 5. Is the model still producing when the client disconnects?

- **In isolation (Ollama only):** no request exceeded ~17 s once loaded; even cold start was ~16 s. At the 45 s mark nothing was still generating in the harness.
- **In production (full server on the 8 GB GPU):** reloads contend with XTTS / Whisper / reranker / embedder for VRAM. `Notepad_Test_Results/test_blue_green.log:4-5` already shows `CUDA out of memory … GPU 0 has a total capacity of 8.00 GiB … 7.84 GiB allocated by PyTorch`. Under that pressure the 7B reload stretches well past 15 s/30 s/45 s. When Assistify's cap fires it is usually because Ollama is **still loading (zero tokens emitted yet)**, or mid-decode of a `num_predict=900` memo. Because the client timeout doesn't cancel the Ollama job, **the model keeps producing after the disconnect**, and the next request collides with it.

So: timeout source = **first-token starvation during model reload**, not slow steady-state decode.

## 8. Candidate fixes (proposed only — not applied)

Ordered by leverage, all model-agnostic and document-agnostic:

1. **Pin a single `num_ctx` for all chat + warmup + grounded calls** (e.g. one value ≥ the largest needed). Eliminates the reload entirely; `num_predict` can still vary safely. This is the single highest-impact change and directly removes the 5–15 s first-token penalty.
2. **Make warmup load the exact `num_ctx` used by generation paths** (or pre-load each distinct `num_ctx` bucket once) so the first real memo/quiz query is warm.
3. **Cancel the in-flight Ollama request on client timeout** (close the response / drop the connection) before falling back, so abandoned generations stop consuming the GPU.
4. **Right-size the cap stack:** the 30 s `gather` + 15 s fallback and the memo 45 s first-token cap should account for a worst-case cold reload, or the reload should be removed (see #1) so 15 s is sufficient.
5. **Reduce VRAM contention** so a reload (if it must happen) is fast — e.g. ensure the LLM keep-alive is honored and other GPU models don't evict it (`OLLAMA_KEEP_ALIVE=5m` env vs `keep_alive=-1` in payloads is itself inconsistent).
